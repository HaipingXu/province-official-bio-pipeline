## 2. Phase 0：官员名单收集（`list_scraper.py`）

### 2.1 目标

构造一份"自 `start_year` 以来在目标城市任职的市长/市委书记"完整名单，保证无遗漏。

### 2.2 四路信息源

| 来源 | 方式 | 可靠性 |
|------|------|--------|
| 百度百科"XX市历任市长"词条 | HTTP 爬取 + 正则提取姓名 | 低（百科列表页经常被反爬拦截或内容不全） |
| 百度百科"XX市历任市委书记"词条 | 同上 | 低 |
| DeepSeek LLM 训练记忆 | 直接问模型列名单（JSON） | 中（可能有错、有遗漏） |
| Qwen LLM 训练记忆 | 同上 | 中 |

**合并逻辑**：四路取并集，姓名仅出现于1路则标记 `needs_check=True`。

### 2.3 关键代码逻辑

**百科列表页爬取**（`scrape_baike_list_page`）
```python
# 尝试三种搜索词：
# "深圳市历任市长"、"深圳历任市长"、"深圳市市长名单"
url = f"https://baike.baidu.com/item/{quote(term)}"
resp = requests.get(url, headers=get_headers(), timeout=15)
# 从页面纯文本中提取 "年份+中文名" 模式
names = extract_names_from_text(text, role, start_year)
```

**姓名提取**（`extract_names_from_text`）
```python
# 找含年份（>= start_year）的行
# 从行中提取 2-4 个汉字的候选名
# 过滤常见非名词（"历任"、"市长"、"广东"…）
name_pattern = re.compile(r"[\u4e00-\u9fff]{2,4}")
```

**LLM 问询**（`llm_get_officials`）
```python
# prompt 要求输出 JSON 数组：
# [{name, role, start_year, end_year, needs_check}]
# temperature=0.1 降低幻觉
response = client.chat.completions.create(
    model=model, messages=[{"role":"user","content":prompt}],
    temperature=0.1, max_tokens=2000
)
```

### 2.4 已知问题

- **百科列表页几乎无效**：百度对列表词条反爬力度更强，`list_scraper` 中的 `requests` 库未使用 curl_cffi，实际命中率很低。四路信息源实际上主要依赖两路 LLM 的记忆。
- **LLM 记忆不可靠**：训练数据截止时间可能导致近年官员遗漏；偶发幻觉（虚构官员）。
- **去重仅靠字符串精确匹配**：同名异人无法区分（如有两位"张伟"先后任职）。
- **`needs_check` 旗标无后续处理**：标记后没有流程提示用户在哪里核查，也不影响后续 pipeline。

## 3. Phase 1：百度百科爬取（`bio_scraper.py`）

### 3.1 目标

从百度百科为每位官员下载完整的个人词条文本，保存为 `officials/{姓名}_biography.txt`。

### 3.2 反爬绕过：curl_cffi

百度百科对普通 `requests` 返回 403，原因是 TLS 握手特征与浏览器不符。

```python
from curl_cffi import requests as cffi_requests

r = cffi_requests.get(
    url,
    impersonate="chrome120",   # 模拟 Chrome 120 的 TLS 指纹
    headers={...},
    timeout=20,
)
```

`curl_cffi` 底层使用 curl 而非 Python 的 `ssl` 模块，能完整复制 Chrome 的 TLS ClientHello，包括 cipher suites 顺序、扩展、JA3 指纹——绕过基于 TLS 指纹的反爬系统。

### 3.3 URL 策略（按优先级）

```python
urls = [
    ("https://baike.baidu.com/item/{name}", False),          # 桌面版（最常命中）
    ("https://m.baike.baidu.com/item/{name}", True),         # 移动版
    ("https://baike.baidu.com/item/{name}（{city}市官员）", False),  # 消歧义版（通常不存在）
]
```

依次尝试，遇到 200 且通过验证则停止。

### 3.4 HTML 解析：`extract_biography_text`

用 BeautifulSoup 分四层提取：

```
① 词条标题（<title>，含当前职务信息）
② 基本信息框（class 含 basic-info / baseInfo）→ 出生日期/籍贯/民族等
③ 正文区（class 含 J-lemma-content / content-wrapper）→ 按 h1-h4/p/li 分节
④ 兜底：全文关键词过滤（含"市长"/"书记"/"1984年"等的行）
```

**问题**：百度百科正在向 React/动态渲染迁移。静态 HTML 中的 class 名经常被混淆（如 `J-lemma-content_bq1la`），CSS 选择器失效，导致层③提取为空，落入层④（关键词过滤），得到的文本量偏少（实测许勤词条仅提取 763 字，而 HTML 原始大小 76,505 字节）。

### 3.5 身份验证：`is_correct_person`

```python
def is_correct_person(text, name, city) -> bool:
    has_city  = city in text or (city+"市") in text
    has_role  = any(kw in text for kw in ["市长","市委书记","省长",...])
    return has_city and has_role
```

若提取的文本不同时包含目标城市和政务职务关键词，则认为爬取到了同名他人（如演员、运动员），跳过该 URL。

### 3.6 已知问题

- **动态渲染内容缺失**：百科现在部分内容由 JS 动态加载，curl_cffi 获取的是首屏 HTML，非 JS 执行后的完整 DOM。履历时间线有时在 JS 区块内，导致提取不完整。
- **身份验证规则偏弱**：仅检查城市名 + 政务关键词，若同名官员曾在别处任职（不是目标城市）也可能通过。
- **无消歧义逻辑**：若百科默认词条是同名知名度更高的人，pipeline 会使用错误词条（除非身份验证拦截）。
- **无重试延迟抖动策略**：403 时固定等 10 秒，容易被检测出机器规律。

## 4. Phase 2：DeepSeek 结构化提取（`api_processor.py`）

### 4.1 目标

将百科纯文本发给 DeepSeek API，按固定 JSON schema 提取结构化履历。

### 4.2 System Prompt 加载（单一信息源原则）

```python
def load_skill_prompt(skill_name: str) -> str:

    # 剥离 YAML frontmatter（--- ... ---）
    body = re.sub(r"^---.*?---\s*", "", content, flags=re.DOTALL)
    return body.strip()

system_prompt = load_skill_prompt("bio-extraction")
```

`.claude/skills/bio-extraction.md` 是唯一权威：修改此文件，DeepSeek/Qwen/Claude 三方同时使用新规则。


### 4.5 增量保存

每处理完一位官员立即写入 `deepseek_results.json`，防止中途报错丢失进度。

### 4.6 已知问题

- **上下文长度限制**：若百科文本过长（如许勤全文超过 50,000 字），可能超出 DeepSeek 单次调用的有效理解范围，导致靠后的履历条目被截断或遗漏。
- **提取质量依赖原文质量**：Phase 1 提取的 763 字文本已足够提取 19 条履历，但若原文在动态区块中，DeepSeek 只能处理有限信息。
- **`升迁/本省提拔/本省学习` 字段可靠性差**：这三个 bio 字段需要 LLM 基于全部履历做综合判断，而非从原文中直接读取——属于推理任务，不同模型、不同运行可能给出不同结论，且没有可验证的来源。
- **组织标签幻觉**：偶发模型使用 30 个以外的自拟标签，目前没有后处理的校验和纠正逻辑。
- **重试机制简单**：仅对网络错误重试，JSON 解析失败不会修改 prompt 后重试（如提示"你输出的 JSON 格式有误"）。

## 5. Phase 3：Qwen 交叉核查（`verifier.py`）

### 5.1 目标

用 Qwen3-max **独立**对同一份百科文本做一次提取，与 DeepSeek 结果对比，自动发现差异。

### 5.2 核查流程

```
① 读取 DeepSeek 结果（deepseek_results.json）
② 对每位官员：
   a. 若 Qwen 缓存已有 → 直接用缓存（跳过重新提取）
   b. 若无缓存 → 调用 Qwen API 提取（同样使用 bio-extraction.md 作 system prompt）
③ 对比 DeepSeek vs Qwen：
   - bio 字段差异（出生年份、籍贯、少数民族、女性、全日制本科）
   - episodes 数量差异
   - 逐条匹配后对比：起始时间、组织标签
④ 输出 verdict：PASS / NEEDS_REVIEW / MAJOR_CONFLICT
⑤ NEEDS_REVIEW 及以上 → 在 Excel 备注栏写入 [需人工核查]
```

### 5.3 Episode 匹配算法（`match_episodes`）

```python
# 用字符集交集相似度匹配"供职单位+职务"
unit_sim = |set(unit_a) & set(unit_b)| / |set(unit_a) | set(unit_b)|
pos_sim  = |set(pos_a)  & set(pos_b)|  / |set(pos_a)  | set(pos_b)|
score = unit_sim * 0.6 + pos_sim * 0.4
# 阈值 0.3，取最高分匹配
```

### 5.4 判决规则

| 条件 | Verdict |
|------|---------|
| HIGH ≥ 2，或 HIGH ≥ 1 且 MEDIUM ≥ 2 | MAJOR_CONFLICT |
| HIGH ≥ 1，或 MEDIUM ≥ 2 | NEEDS_REVIEW |
| 其余 | PASS |

HIGH 触发条件：出生年份/少数民族/性别不一致、日期差 > 2 年、条数差 > 4
MEDIUM 触发条件：组织标签不同、日期差 > 1 年、条数差 > 2

### 5.5 已知问题

- **核查结论不被使用**：当前 Phase 4 仅将核查结论写入备注栏文字，不会触发任何自动纠正或"取 Qwen 结果覆盖 DeepSeek"的逻辑。换言之，即使 MAJOR_CONFLICT，输出数据仍完全来自 DeepSeek。
- **缓存污染**：若 Phase 1 爬到了错误数据（如 LLM 生成的假传记），Qwen 缓存里存的也是基于错误数据的结果，`--force` 清楚了 DeepSeek 缓存但可能漏掉 Qwen 缓存。
- **匹配算法脆弱**：字符集交集相似度对于机构名很短（如"省委"）或职务写法差异大（"市委书记" vs "书记"）时会失配，产生误报差异。
- **同一官员同时期的两条记录**：若 DeepSeek 拆行方式与 Qwen 不同（如一条 vs 两条），差异计数会虚高，触发 MAJOR_CONFLICT 但实际数据都正确。

## 6. Phase 4：后处理（`postprocess.py`）

### 6.1 目标

将 `deepseek_results.json`（嵌套 JSON）扁平化为逐行 dict，计算所有派生列。

### 6.2 扁平化逻辑

```python
for result in deepseek_results:
    bio      = result["bio"]       # 人员级别信息
    episodes = result["episodes"]  # 履历列表

    # 先计算行级标志（Z/AA）
    for ep in episodes:
        is_mayor_row = is_shenzhen_mayor_row(ep["职务"], ep["供职单位"], city)
        is_sec_row   = is_shenzhen_secretary_row(ep["职务"], ep["供职单位"], city)

    # 再计算人员级别标志（M/N）
    ever_mayor = int(any(r["is_mayor_row"] for r in row_data))
    ever_sec   = int(any(r["is_sec_row"]   for r in row_data))

    # 逐行输出
    for ep in episodes:
        row = {所有列...}
```

### 6.3 关键判断函数

**市长行识别**（`is_shenzhen_mayor_row`）
```python
is_mayor = (
    ("市长" in position or "代市长" in position)
    and "副市长" not in position        # 排除副市长
    and "常务副" not in position         # 排除常务副市长
    and "第一副" not in position         # 排除第一副市长
)
city_in_unit = (
    city in unit or f"{city}市" in unit
    or "市政府" in unit or "市人民政府" in unit
)
```

**书记行识别**（`is_shenzhen_secretary_row`）
```python
sec_keywords = ["市委书记", "委书记"]
if kw in position and (city in unit or "市委" in unit): return 1
# 特殊处理：职务="书记" 且 单位="{city}市委"
```

### 6.4 已知问题

- **"年份"列（A列）逻辑不一致**：`shenzhen_start_year` 取的是该官员在深圳任职（市长/书记）的最早起始年份，同一官员所有行都用这一个年份。但若一个人先当市长后当书记，A 列年份是市长任职年份，不是书记任职年份，可能与研究者期望不同。
- **`备注栏` 核查标记只写在第0行**：`verif_note if (verif_note and i == 0) else ep.get("备注栏","")` ——仅第一条履历有标记，其余行看不出此人有问题。
- **`升迁/本省提拔/本省学习` 无后处理校验**：完全信任 DeepSeek 输出，没有基于 episodes 数据做规则校验（例如可以从 episodes 中最后一条职务推算升迁）。
- **`任职地` 字段不统一**：DeepSeek 有时输出"广东"，有时"广东省"，有时"深圳"，没有标准化逻辑。

```python
for col_cells in ws.columns:
```

## 8. 提取规则（单一信息源：`.claude/skills/bio-extraction.md`）

### 8.1 输出 Schema

```json
{
  "bio": {
    "姓名": "string",
    "出生年份": int,
    "籍贯": "XX省",
    "少数民族": 0|1,
    "女性": 0|1,
    "全日制本科": 0|1,
    "升迁": 0|1,
    "本省提拔": 0|1,
    "本省学习": 0|1
  },
  "episodes": [{
    "经历序号": int,
    "起始时间": "YYYY.MM",
    "终止时间": "YYYY.MM 或 至今",
    "组织标签": "30选1",
    "供职单位": "string",
    "职务": "string",
    "任职地": "string",
    "中央/地方": "中央|地方",
    "时长": "float string",
    "是否落马": "是|否",
    "备注栏": "string"
  }]
}
```

### 8.2 核心规则摘要

| 规则 | 说明 |
|------|------|
| 行拆分 | 不同机构兼任 → 分行；同机构兼任 → 合并；晋升 → 分行 |
| 组织标签 | 29个固定分类+其他，禁止自拟 |
| 时间格式 | YYYY.MM；无月份写 YYYY.00；在职写"至今" |
| 中央/地方 | 中央机关=中央，其余=地方（北京市机关算地方） |
| 信息源约束 | 严格以百度百科为准，不得凭想象填写 |

## 10. 优化方向建议

### 10.1 Phase 0：提高名单完整性

**方案A（推荐）**：引入第三方来源
- 中央组织部官网 / 各城市政府官网的历届领导人页面
- Wikipedia（各城市"历任市长"词条通常比百度更完整）
- 人民网、新华网的届次报道

**方案B**：改进百科爬取
- 使用 Selenium 或 Playwright 执行 JavaScript，获取动态渲染的列表内容
- 可与 curl_cffi 组合：先用 cffi 尝试，超时/空内容则启动浏览器引擎

**方案C**：人工维护基准名单
- 对于每个城市，维护一份 `data/{city}_known_officials.csv`
- Pipeline 以此为基准，LLM 仅补充基准中没有的人
