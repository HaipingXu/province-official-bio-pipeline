## 2. Phase 0：人工名单解析（input_parser.py）

### 2.1 设计动机

v1 使用 LLM 自动从百度百科列表页抓取官员名单，存在以下问题：
- 百度百科列表页结构不稳定，经常返回403或缺少信息
- LLM 可能漏掉代市长等特殊任期
- 无法控制任期起止时间的精度

v2/v3 改为**人工维护 txt 文件**，格式简单清晰，完全可控。

### 2.2 文件格式

```
城市：深圳
省份：广东
起始年份：2010
维基列表_市长：https://zh.wikipedia.org/zh-sg/深圳市市长列表

[市长]
许勤, 2010.06-2017.01
覃伟中（代）, 2021.04-2021.05    ← （代）标记代市长
覃伟中, 2021.05-至今             ← 至今表示在任

[市委书记]
王荣, 2010.04-2015.03
```

### 2.3 解析逻辑

- `parse_entry()` 提取姓名、任期起止、是否代理
- `parse_officials_txt()` 读取全部市长+书记，去重合并到 `all_officials`
- `compare_with_wiki()` 可选：抓取维基百科列表页对比，报告差异（返回 `list[str]`）

## 3. Phase 1：两层爬取策略（bio_scraper_v2.py）

### 3.1 两层策略

| 层 | 工具 | 触发条件 | 预期成功率 |
|----|------|---------|-----------|
| 1 | curl_cffi（Chrome120 TLS伪装）| 默认 | ~70% |
| 2 | Playwright + playwright-stealth | Layer1 文本 < 500字 | ~92% |

### 3.2 并发爬取（v3新增）

```python
def scrape_all(officials, city="", force=False, max_workers=SCRAPE_WORKERS):
    # max_workers 默认2，防止反爬检测
    with ThreadPoolExecutor(max_workers=max_workers) as pool:
        futures = {pool.submit(scrape_official, o["name"], city, force): o for o in officials}
        for future in as_completed(futures):
            ok, result = future.result()
```

- 每个官员独立爬取（独立文件输出），天然线程安全
- Playwright 浏览器实例共享但 context 按调用创建
- `close_browser()` 在所有 futures 完成后调用
- `--workers 1` 回退串行模式

### 3.3 内容提取（v3.1重写）

百度百科使用 CSS-module 哈希类名（如 `lemmaSummary_RPxMM`、`basicInfo_o5Omt`），
**不能**用精确 class 匹配。v3.1 采用**前缀关键词匹配** + **7层提取策略**：

```python
def _match_class(cls_list, *keywords):
    """检查 CSS class 是否包含任一关键词前缀（不区分大小写）"""
    joined = " ".join(cls_list).lower()
    return any(kw in joined for kw in keywords)
```

7 层提取顺序：

| 层 | 目标 | 匹配方式 |
|----|------|---------|
| 0 | 移除 `lemmaReference` 参考文献区域 | `_match_class("lemmaref", "reference")` → `decompose()` |
| 1 | 词条标题 | `<title>` 标签 |
| 2 | 人物简介 | `_match_class("lemmasummary")` → 子 `<div>` 段落 |
| 3 | 基本信息卡 | `_match_class("basicinfo")` 或 `id="J-basic-info"` |
| 4 | 正文段落（主要来源）| `id="J-lemma-content"` → `_match_class("para")` 子 div |
| 5 | 履历段落回退 | 内容启发式（年份模式 + 职务关键词） |
| 6 | 任期表格 | 含 `任期`/`职务`/`起始` 表头的 `<table>` |
| 7 | 纯文本回退 | 按行过滤含年份的行 |

关键改进：
- **参考文献移除**：在提取前清除 `lemmaReference` div，防止参考文献中的年份和职务关键词污染履历文本
- **引注标记清理**：正则移除 `[1]`、`[2]` 等百科引注标记
- **`min_chars` 阈值降至 500**（原 2000），适配简历较短的官员（如孟凡利 ~1570字）

### 3.4 人物核验

`is_correct_person(text, name, city)` 检查：
- 文本中包含目标城市名（支持市/州/盟/地区等行政后缀）
- 文本中包含官员职务关键词

## 4. Phase 2：DeepSeek两步提取（api_processor_v2.py）

### 4.1 两步架构设计

| | Step 1 | Step 2 |
|--|--------|--------|
| 任务 | 提取履历事实 | 分析三个标签 |
| 输入 | 百科原文 | Step1 episodes 列表 |
| 输出 | raw_bio + episodes | 升迁/本省提拔/本省学习（含依据） |
| Prompt | bio-extraction-step1.md | bio-labeling.md |
| 关键字段 | 原文引用（15-50字） | 每标签必须写依据 |

### 4.2 并发提取（v3新增）

```python
def process_all_officials(..., max_workers=DEFAULT_WORKERS):
    lock = threading.Lock()

    def _process_one(official):
        # Step1 → Step2 串行（Step2 依赖 Step1）
        s1 = step1_extract(...)
        with lock:
            existing_s1[name] = s1
            _save_cache(step1_path, existing_s1)  # 增量保存
        s2 = step2_label(..., s1["episodes"])
        with lock:
            existing_s2[name] = s2
            _save_cache(step2_path, existing_s2)

    with ThreadPoolExecutor(max_workers=max_workers) as pool:
        futures = [pool.submit(_process_one, o) for o in officials]
```

- **同一官员内** Step1→Step2 串行（Step2 依赖 Step1 结果）
- **不同官员间** 并发执行
- `threading.Lock` 保护共享缓存 dict 和 JSON 文件写入
- OpenAI client 内部基于 httpx，线程安全

### 4.3 增量保存

每处理完一个官员立即保存结果（加锁），防止中途失败丢失进度。

### 4.4 API 调用工具（utils.py）

所有 LLM 调用通过 `llm_chat()` 统一处理：
- 自动重试（429 指数退避，其他错误线性退避）
- 支持 `extra_body` 参数（Qwen 的 `enable_thinking: False`）
- JSON 提取通过 `extract_json()` 自动剥离 ```json 围栏

## 5. Phase 3a：Qwen独立核查（verifier_v2.py）

### 5.1 完全独立运行

Qwen 运行相同的 Step1+Step2 提取，完全不看 DeepSeek 的结果，保证验证独立性。

### 5.2 并发核查（v3新增）

```python
def run_verification(..., max_workers=DEFAULT_WORKERS):
    lock = threading.Lock()

    def _verify_one(name, ds_s1):
        qw_s1 = qwen_step1(...)
        with lock: qw1_cache[name] = qw_s1; _save_cache(...)
        qw_s2 = qwen_step2(...)
        with lock: qw2_cache[name] = qw_s2; _save_cache(...)
        diffs = diff_bio_fields(...) + diff_label_fields(...) + diff_episodes(...)
        return {"official_name": name, "verdict": ..., "diffs": diffs}

    with ThreadPoolExecutor(max_workers=max_workers) as pool:
        futures = {pool.submit(_verify_one, n, s1): n for n, s1 in ds_step1.items()}
```

### 5.3 Episode 匹配算法

DS 和 QW 可能对同一人提取出不同数量的 episodes，需要先匹配对应行：

```
相似度 = 原文引用相似度(50%) + 供职单位相似度(30%) + 职务相似度(20%)
阈值 = 0.25，低于阈值视为"无对应行"
```

### 5.4 差异分类与判定

| 类型 | 说明 |
|------|------|
| `bio` | raw_bio 字段差异（出生年份、籍贯等） |
| `episode_count` | 两方提取条数差 > EPISODE_COUNT_DIFF |
| `episode_missing` | 一方有一方无的整条 |
| `episode_field` | 同条内字段值不同 |
| `label` | 升迁/本省提拔/本省学习值不同 |

判定规则：
- HIGH ≥ 2 或 (HIGH ≥ 1 且 MEDIUM ≥ 2) → `MAJOR_CONFLICT`
- HIGH ≥ 1 或 MEDIUM ≥ 2 → `NEEDS_REVIEW`
- 其他 → `PASS`

## 6. Phase 3b：Battle表+Kimi K2.5裁判（battle_generator.py）

### 6.1 裁判策略（v3更新，v3.1修复）

- **裁判模型**：Kimi K2.5（`api.moonshot.cn`），后备 DeepSeek
- **仅争议字段**调用裁判（DS ≠ QW），避免不必要的 API 调用
- **裁判缓存**：`logs/judge_decisions.json`，重跑时不重复调用

**Kimi K2.5 API 约束**（推理模型，v3.1发现并修复）：

| 约束 | 说明 |
|------|------|
| `temperature` | **只能为 1**（传其他值返回 HTTP 400：`invalid temperature`） |
| `max_tokens` | 需 ≥ 4096（推理模型用 token 做内部思考，200 会耗尽 budget 返回空内容） |
| `response_format` | **不支持** structured output（Moonshot API 限制） |
| 重试策略 | 因 temperature=1 输出随机性大，每次裁判调用最多重试 2 次 |
| 后备模型 | 2次均失败/JSON解析失败 → 回退 DeepSeek（temperature=0） |

```python
is_kimi = "kimi" in _judge_model.lower()
judge_temp = 1.0 if is_kimi else 0.0
max_retries = 2 if is_kimi else 1
# max_tokens=4096 确保推理后还有余量输出 JSON
```

### 6.2 并发裁判（v3新增）

v3 采用**批量收集+并发执行**策略：

```python
# 1. 先收集所有待裁判项
pending_calls = []
for row in episode_rows:
    for f in disputed_fields:
        if cache_key not in judge_cache:
            pending_calls.append((cache_key, kwargs))

# 2. 批量并发调用
with ThreadPoolExecutor(max_workers=max_workers) as pool:
    futures = {pool.submit(_judge_one, ck, kw): ck for ck, kw in pending_calls}
    for f in as_completed(futures):
        ck, decision = f.result()
        with lock: judge_cache[ck] = decision

# 3. 回填裁判结果到行
for row in episode_rows:
    for f in disputed_fields:
        decision = judge_cache[cache_key]
        ...
```

## 7. Phase 4：后处理（postprocess_v2.py）

### 7.1 地名标准化

LLM 经常输出省份简称，通过 `config.PROVINCE_NORMALIZE` / `CITY_NORMALIZE` 统一：

```python
"广东" → "广东省"
"北京" → "北京市"
"内蒙古" → "内蒙古自治区"
```

### 7.2 市长/书记标志位判断

**`该条是市长`（AB列）** = 1 当且仅当：
- 职务包含 `市长` 或 `代市长`
- **且**不包含 `副市长`、`常务副`
- **且**供职单位/任职地匹配目标城市（使用 `_strip_admin_suffix()` 处理市/州/盟/地区）

**`该条是书记`（AC列）** = 1 当且仅当：
- 职务包含 `市委书记`
- **且**不包含 `副书记`、`市委副书记`
- **且**供职单位/任职地匹配目标城市

### 7.3 裁判结果回流（v3改进）

`_load_judge_overrides()` 从 `logs/judge_decisions.json` 加载裁判决定，将 `采纳QW` 的字段值覆盖到最终输出。

### 7.4 组织标签校验

不在 30个标签之内的值 → 标记为 `[无效标签]{原值}`，便于人工核查。

## 9. 并发处理架构

### 9.1 设计原则

| 原则 | 实现 |
|------|------|
| 方案 | `ThreadPoolExecutor`（非 asyncio），改动最小 |
| 线程安全 | `threading.Lock` 保护共享 dict + JSON 文件写入 |
| 可配置 | `--workers N` CLI 参数，默认 3 |
| 爬取限速 | 爬取阶段最多 `min(workers, 2)` 防反爬 |
| 增量保存 | 每个 worker 完成后立即写入 JSON（加锁） |
| 串行回退 | `--workers 1` 完全等价于原串行行为 |

### 9.2 各阶段并发模型

| 阶段 | 并行单元 | 最大workers | 串行依赖 |
|------|---------|------------|---------|
| Phase 1 爬取 | 每官员 | min(workers, 2) | 无 |
| Phase 2 提取 | 每官员 | workers | 同一官员 Step1→Step2 |
| Phase 3a 核查 | 每官员 | workers | 同一官员 qwen_step1→step2→diff |
| Phase 3b 裁判 | 每裁判调用 | workers | 先收集全部争议→批量并发 |
| Phase 4-5 | 无 | - | 串行处理 |

### 9.3 线程安全保障

```python
lock = threading.Lock()

# 读缓存（检查是否已处理）
with lock:
    cached = cache.get(name)

# 写缓存（处理完成后）
with lock:
    cache[name] = result
    path.write_text(json.dumps(list(cache.values()), ...))
```

关键点：
- OpenAI client (`httpx`) 线程安全，可跨线程共享
- Playwright 浏览器实例共享但 context 按调用创建
- JSON 文件写入在锁内完成，不会出现写截断

### 9.4 性能预估

典型场景（10个官员）：

| 阶段 | 串行耗时 | 3 workers | 提速比 |
|------|---------|-----------|-------|
| Phase 1 爬取 | ~60s | ~30s (2w) | 2x |
| Phase 2 提取 | ~60s | ~20s | 3x |
| Phase 3a 核查 | ~60s | ~20s | 3x |
| Phase 3b 裁判 | ~30s | ~10s | 3x |
| **总计** | **~210s** | **~80s** | **~2.6x** |

### v3.1 Bug 修复（2026-03-19）
